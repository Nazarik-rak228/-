## **ISO/IEC 12207 — Software Life Cycle Processes**

### Цель стандарта

Определить **процессы жизненного цикла программного обеспечения** от идеи до вывода из эксплуатации. Стандарт задаёт общий язык для заказчиков, разработчиков и тестировщиков.
### Основная структура

Стандарт описывает группы процессов:

- **Основные процессы** (разработка, сопровождение, эксплуатация)
- **Вспомогательные процессы** (документирование, верификация, валидация, тестирование)
- **Организационные процессы** (управление, улучшение, обучение)
### Ключевые требования и рекомендации
- Чёткое разделение этапов жизненного цикла ПО
- Наличие формализованных требований и документации
- Проведение верификации и валидации на всех этапах
- Тестирование рассматривается как обязательная часть жизненного цикла, а не финальный шаг
### Значение для разработки и тестирования
Стандарт помогает:
- организовать процесс разработки;
- встроить тестирование на каждом этапе;
- контролировать качество продукта системно.

## 2. **ISO/IEC 25010 — System and Software Quality Models**

### Цель стандарта

Определить **модель качества программного продукта** и дать критерии его оценки.
### Основная структура

Стандарт включает две модели:
1. **Качество продукта**
2. **Качество при использовании**

Основные характеристики качества ПО:

- функциональная пригодность
- надёжность
- производительность
- удобство использования
- совместимость
- безопасность
- сопровождаемость
- переносимость
### Ключевые требования и рекомендации

- Качество ПО должно оцениваться по нескольким характеристикам
- Требования к качеству должны быть измеримыми
- Тестирование должно проверять не только функциональность, но и нефункциональные характеристики

### Значение для разработки и тестирования

Используется для:

- формирования требований к качеству;
- планирования тестов;
- оценки готовности продукта к выпуску.

## 3. **ISO/IEC/IEEE 29119 — Software Testing**

### Цель стандарта

Установить **единые международные правила и терминологию тестирования ПО**.

### Основная структура

Стандарт состоит из нескольких частей:

- концепции и термины тестирования
- процессы тестирования
- документация тестирования
- методы и техники тестирования
    

### Ключевые требования и рекомендации

- Наличие стратегии и плана тестирования
- Формализация тест-кейсов и отчётов
- Связь тестирования с требованиями
- Использование различных уровней тестирования (модульное, интеграционное, системное)
    

### Значение для разработки и тестирования

Стандарт:

- повышает прозрачность тестирования;
- помогает стандартизировать тестовую документацию;
- облегчает взаимодействие команд.

## 4. **IEEE 829 (устаревший, но широко используемый) — Test Documentation**

### Цель стандарта
Определить **набор документов для процесса тестирования**.
### Основная структура

Документы тестирования:
- план тестирования
- описание тестового дизайна
- тест-кейсы
- отчёты о дефектах
- итоговый отчёт о тестировании
### Ключевые требования и рекомендации

- Каждый этап тестирования должен быть задокументирован
- Документы должны быть понятны всем участникам проекта
- Результаты тестирования должны быть отслеживаемыми
### Значение для разработки и тестирования

Хотя стандарт заменён ISO/IEC 29119, он до сих пор используется:

- в учебных проектах;
- в компаниях со строгой отчётностью;
- как основа для тестовой документации.
