###   я выступаю за введение CI/CD, вот аргументы

Внедрение CI для не-ядровых модулей в государственной системе электронного голосования целесообразно. Это автоматизирует интеграцию изменений через Merge Request с запуском пайплайна для сборки и тестирования, без развертывания, что соответствует приоритетам стабильности и качества в монолите на Java, не затрагивая продакшн.

- **Раннее выявление проблем и повышение качества кода**: CI обнаруживает баги на ранних стадиях через автоматические тесты при создании Merge Request. В системе с обновлениями раз в полгода и ручной проверкой это сократит время на поиск ошибок, предотвращая накопление технического долга в кодовой базе.
- **Быстрая обратная связь для команды**: Разработчики получают уведомления об ошибках моментально, ускоряя исправления. Для команды из 40 человек это повысит эффективность, мотивируя энтузиастов и позволяя специалистам по кибербезопасности фокусироваться на критических задачах, без нарушения регламентов.
- **Автоматизация рутины с сохранением стабильности**: CI не включает развертывание, поэтому сборка остается под контролем администраторов на выделенных серверах. Это упрощает поддержку стабильности, исключая не прошедшие тесты изменения, что идеально для изолированных модулей и соответствует ФСТЭК.
- **Обмен знаниями и улучшение командной работы**: Через code review в Merge Request команда делится знаниями о проекте, повышая общую продуктивность и прозрачность, что снижает конфликты в смешанной команде.
- **Гибкость и экономия ресурсов**: Автоматизация тестов и сборки сокращает время релизов и человеческих ошибок. В 7-летней системе это позволит адаптироваться к новым требованиям, освобождая ресурсы для ядра, без компромисса безопасности.

### Поэтапное и безопасное внедрение элементов CI/CD

План внедрения начинается с CI для не-ядровых модулей, с изоляцией в контуре и согласованиями ФСТЭК. Начать с линтеров и юнит-тестов. Общий timeline — 8-15 месяцев, с постепенным rollout и мониторингом на каждом шаге для корректировки.

#### Этап 1: Анализ текущих процессов и формирование группы (1 месяц)

- Провести аудит существующих workflow: выявить bottlenecks в ручной проверке и тестировании. Собрать фидбек от команды.
- Сформировать межфункциональную группу (разработчики, аналитики, специалисты по безопасности, администраторы) для планирования.
- Безопасность: Определить риски, связанные с новыми инструментами, и подготовить предварительный план изоляции.

#### Этап 2: Подготовка документации и согласование (1-2 месяца)

- Разработать детальную документацию: описание CI (Merge Request запускает конфигурацию с этапами сборки и тестирования), преимущества для качества кода.
- Согласовать с ФСТЭК, акцентируя изоляцию от продакшн и соответствие регламентам.
- Безопасность: Self-hosted runners в контуре, теги для контроля доступа.

#### Этап 3: Установка и настройка инфраструктуры (1 месяц)

- Установить GitLab в изолированном окружении, настроить репозитории для не-ядровых модулей.
- Настроить базовые runners с ограниченным доступом, интегрировать с существующими серверами.
- Безопасность: Тестирование на уязвимости, установка ролевых доступов.

#### Этап 4: Миграция кода и базовый CI (1-2 месяца)

- Перенести не-ядровые модули в GitLab, создать конфигурацию (.gitlab-ci.yml).
- Применить сначала: линтеры (для проверки синтаксиса в этапе сборки), юнит-тесты (в этапе тестирования).
- Безопасность: Запуск только при Merge Request, ручное одобрение после CI.

#### Этап 5: Написание и интеграция тестов (1-2 месяца)

- Разработать или адаптировать существующие тесты для Java (JUnit для юнит-тестов).
- Добавить скрипты для подготовки (before_script) и очистки (after_script).
- Безопасность: Внедрить статический анализ на уязвимости в пайплайн.

#### Этап 6: Расширение пайплайна и дополнительные проверки (1-2 месяца)

- Добавить интеграционные тесты для взаимодействия модулей, переменные для конфигураций.
- Интегрировать уведомления и code review в процесс.
- Безопасность: Хранение артефактов (результатов тестов), аудит логов и кэширование для ускорения без рисков.

#### Этап 7: Обучение команды и пилотный запуск (1 месяц)

- Провести тренинги по Merge Request, пайплайну и новым workflow для всех 40 человек.
- Запустить CI на пилотном модуле, мониторить метрики (время на MR, количество багов).
- Безопасность: Симуляция сценариев сбоев в контуре.

#### Этап 8: Интеграция в повседневный процесс (1 месяц)

- Сделать CI обязательным для всех новых Merge Request в не-ядровых модулях.
- Собрать фидбек и внести корректировки в конфигурацию.
- Метрики: Снижение ошибок, ускорение проверок.

#### Этап 9: Оценка результатов и оптимизация (1 месяц)

- Проанализировать общие метрики: влияние на качество кода, мотивацию команды.
- Оптимизировать пайплайн (добавить rules для условий запуска, parallel для ускорения).
- Безопасность: Повторный аудит с ФСТЭК.

#### Этап 10: Введение элементов CD (если успешно, 2-3 месяца)

- Добавить этап развертывания для staging-окружения с ручным триггером, без продакшн.
- Интегрировать environment для тестовых сред.
- Безопасность: Переменные для sensitive данных, interruptible для экономии ресурсов.
